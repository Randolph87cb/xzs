# 智能训练与题目质量审核方案

## 背景

当前智能训练配置按知识点固定题数抽题，例如某个知识点固定抽 1 题。对于低频知识点，这会导致它们在每次训练中稳定出现，实际效果上把“不常见知识点”变成了“常见知识点”。

题库质量侧还需要补充老师审核能力：老师可以审核并修改题目解析、知识点配置，以及学生错题改错内容。题目解析和知识点审核以“老师修订为可信内容”为核心，保留审核人、审核时间、审核前后内容和自动推导的审核轮次。学生错题改错审核则需要区分“通过 / 不通过”：通过后错题本不再提示继续改错，不通过后回到错题本继续显示为被驳回状态，学生可重新提交。

## 目标

- 智能训练支持每个知识点配置抽题下限和上限。
- 低频知识点可配置为 `0-1`，避免每次训练都出现。
- 智能训练抽题在满足总题数的前提下，按知识点分布和权重随机分配。
- 老师可以审核并修改题目解析。
- 老师可以审核并修改题目知识点。
- 老师审核知识点时能看到该等级下全部知识点分布。
- 老师可以选择已有知识点，也可以新增知识点。
- 审核记录支持一审、二审等多轮记录。
- 解析审核支持 Markdown 编辑与实时预览。
- 题目质量审核支持按审核状态筛选，能直接查看未审核题目。
- 学生错题本支持提交改错说明，并按未提交、待审核、已通过、被驳回分层展示。
- 老师审核学生改错时选择通过或不通过；不通过后学生可重新提交。

## 非目标

- 不要求学生端参与题目解析或知识点审核。
- 不在第一阶段重构完整题库模型。
- 不强制引入工作流引擎，先用明确状态字段和审核记录表实现。
- 题目解析和知识点审核不做“驳回 / 重新提交”流程，老师直接修订内容。
- 错题改错不做多轮审核轮次，只保留每次老师审核的结论和记录。

## 一、智能训练上下限配置

### 配置结构

将当前规则：

```json
{
  "knowledgePoint": "GESP1级/循环结构",
  "questionCount": 5
}
```

升级为：

```json
{
  "knowledgePoint": "GESP1级/循环结构",
  "minCount": 2,
  "maxCount": 6,
  "weight": 105,
  "enabled": true
}
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `knowledgePoint` | 知识点名称，继续使用 `GESP{N}级/知识点` 格式 |
| `minCount` | 每次训练至少抽几题，可为 0 |
| `maxCount` | 每次训练最多抽几题 |
| `weight` | 分配剩余题数时使用的权重，默认可取该知识点题库数量 |
| `enabled` | 是否启用该知识点 |

兼容规则：

- 如果旧配置只有 `questionCount`，则读取时视为 `minCount = questionCount`、`maxCount = questionCount`。
- 保存后统一写入新结构。

### 抽题算法

1. 读取当前科目智能训练配置。
2. 过滤 `enabled = false` 的知识点。
3. 校验每条规则：`0 <= minCount <= maxCount`。
4. 先按 `minCount` 为每个知识点分配基础题数。
5. 计算剩余题数：`remaining = totalQuestionCount - sum(minCount)`。
6. 在未达到 `maxCount` 的知识点中，按 `weight` 加权随机分配剩余题数。
7. 如果某知识点可用题数不足，按实际可用题数降级，并将缺口重新分配给其他知识点。
8. 如果所有可用题都不足以满足总题数，返回明确错误。
9. 每个知识点内仍按随机题目抽取。

### 默认配置建议

按知识点分布生成默认上下限：

| 分布数量 | 建议区间 |
| ---: | --- |
| 1-3 | `0-1` |
| 4-10 | `0-2` |
| 11-30 | `1-3` |
| 31-80 | `1-5` |
| 80 以上 | `2-6` |

这样低频知识点不会每次出现，高频知识点仍能保持更高出现概率。

### 涉及改动

- 后端 `SmartTrainingRuleVM` 增加 `minCount`、`maxCount`、`weight`、`enabled`。
- 后端智能训练抽题逻辑从固定题数改为上下限加权分配。
- 后端配置校验从“规则题数合计等于总题数”改为：
  - `sum(minCount) <= totalQuestionCount`
  - `sum(maxCount) >= totalQuestionCount`
  - 每条规则 `minCount <= maxCount`
- 管理端智能训练配置页增加上下限、权重、启用开关。
- SQL 默认配置从固定题数 JSON 改为上下限 JSON。

## 二、题目解析与知识点审核

### 核心原则

题目解析和知识点审核不是“通过 / 驳回”，而是“老师把内容修订到可用状态”。每次审核都保存审核前内容、审核后内容、审核人和轮次。

审核轮次不让老师手动填写。后端保存时按历史记录自动计算：

```text
nextRound = max(review_round where question_id = ? and review_type = ?) + 1
```

如果没有历史记录，则本次为一审；已有一审记录，则本次自动为二审。前端只展示“本次将保存为一审 / 二审 / 三审”，不提供可编辑轮次输入框。

### 新增表：题目审核记录

建议新增 `t_question_review_record`：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `question_id` | 题目 ID |
| `review_type` | `ANALYSIS` 或 `KNOWLEDGE_POINT` |
| `review_round` | 审核轮次，例如 1、2 |
| `before_value` | 审核前内容 |
| `after_value` | 审核后内容 |
| `reviewer_id` | 审核老师 ID |
| `reviewer_name` | 审核老师姓名 |
| `review_comment` | 审核说明 |
| `create_time` | 记录创建时间 |

题目表可增加质量状态字段：

| 字段 | 含义 |
| --- | --- |
| `analysis_review_round` | 解析已完成的最高审核轮次 |
| `analysis_review_user_id` | 最近解析审核人 |
| `analysis_review_time` | 最近解析审核时间 |
| `knowledge_review_round` | 知识点已完成的最高审核轮次 |
| `knowledge_review_user_id` | 最近知识点审核人 |
| `knowledge_review_time` | 最近知识点审核时间 |

不增加 `REJECTED` 状态。需要区分时使用：

- `UNREVIEWED`：未审核。
- `REVIEWED_ONCE`：一审完成。
- `REVIEWED_TWICE`：二审完成。

这些状态可由审核轮次推导，也可以冗余存储便于筛选。

### 解析审核

老师页面展示：

- 题干。
- 选项。
- 正确答案。
- 当前解析。
- 可编辑解析。
- Markdown 预览。
- 审核历史。

老师操作：

- 编辑解析。
- 填写审核说明。
- 查看 Markdown 实时预览。
- 保存审核结果，轮次由后端自动推导。

保存后：

- 将修订后的解析写回题目内容 JSON。
- 写入 `t_question_review_record`。
- 更新题目解析审核轮次和审核人。

Markdown 预览建议：

- 编辑区和预览区左右分栏；窄屏或弹窗宽度不足时上下排列。
- 预览区复用现有 `QuestionMarkdown` 渲染组件，保证公式、代码块和题目内容展示风格一致。
- 保存前预览区必须展示当前编辑内容，而不是上一次已保存内容。
- 历史记录中 `before_value`、`after_value` 可提供“查看预览”按钮，避免长 Markdown 直接撑开表格。

### 题目质量审核筛选

题目质量审核页需要能直接定位未审核题目，筛选条件至少包括：

| 筛选项 | 含义 |
| --- | --- |
| `subjectId` | 科目 / 等级 |
| `reviewType` | `ANALYSIS` 或 `KNOWLEDGE_POINT` |
| `reviewStatus` | `UNREVIEWED`、`REVIEWED_ONCE`、`REVIEWED_TWICE`、`REVIEWED_AT_LEAST_ONCE` |
| `knowledgePoint` | 当前知识点 |
| `keyword` | 题干关键字 |

列表展示建议：

- 默认筛选 `UNREVIEWED`，让老师进入页面后优先处理未审核题目。
- 同一题同时展示解析审核轮次和知识点审核轮次。
- 未审核题目使用明显状态标签，例如“解析未审”“知识点未审”。
- 已审核题目显示最近审核人、最近审核时间和最高轮次。

后端筛选逻辑：

- 未审核解析：不存在 `review_type = 'ANALYSIS'` 的审核记录。
- 未审核知识点：不存在 `review_type = 'KNOWLEDGE_POINT'` 的审核记录。
- 一审完成：对应类型 `max(review_round) = 1`。
- 二审完成：对应类型 `max(review_round) >= 2`。
- 至少审核一次：对应类型存在任意审核记录。

### 知识点审核

老师页面展示：

- 题目内容。
- 当前知识点。
- 当前等级下全部知识点分布数量。
- 已有知识点选择器。
- 新增知识点输入。
- 审核历史。

老师操作：

- 选择已有知识点。
- 或新增知识点。
- 保存审核结果，轮次由后端自动推导。

保存后：

- 将修订后的知识点写回 `t_question.knowledge_point`。
- 写入 `t_question_review_record`。
- 更新题目知识点审核轮次和审核人。

### 知识点分布接口

新增或扩展后台接口：

```text
POST /api/admin/question/knowledgePointDistribution/{subjectId}
```

返回：

```json
[
  {
    "knowledgePoint": "GESP1级/循环结构",
    "questionCount": 105,
    "reviewedCount": 80,
    "unreviewedCount": 25
  }
]
```

用途：

- 审核知识点时提供全量分布参考。
- 智能训练配置时也可复用。

## 三、错题改错审核

### 核心原则

学生提交改错内容后，老师审核结论分为“通过”和“不通过”。错题改错不需要审核轮次，也不需要一审、二审概念。

- 通过：说明学生改错质量合格，错题本展示为“改错已通过”，不再提示继续提交。
- 不通过：说明学生改错质量不合格，错题本展示为“改错被驳回”，学生可基于老师意见重新提交。
- 待审核：学生已经提交，但老师尚未处理。
- 未提交：错题尚未提交改错。

老师审核时可以填写审核意见，也可以在必要时补充建议，但不把老师修订版作为最终替代文本；错题改错的主体仍应是学生自己的错误原因和正确思路。

### 新增表：错题改错记录

建议新增 `t_question_correction_record`：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `user_id` | 学生 ID |
| `question_id` | 题目 ID |
| `exam_paper_answer_id` | 试卷作答 ID |
| `customer_answer_id` | 题目作答 ID |
| `student_wrong_reason` | 学生提交的错误原因 |
| `student_correct_thinking` | 学生提交的正确思路 |
| `review_status` | `NOT_SUBMITTED`、`SUBMITTED`、`APPROVED`、`REJECTED` |
| `reviewer_id` | 最近审核老师 ID |
| `reviewer_name` | 最近审核老师姓名 |
| `review_comment` | 老师审核说明 |
| `submit_time` | 学生提交时间 |
| `review_time` | 最近审核时间 |
| `resubmit_count` | 重新提交次数 |
| `deleted` | 软删除标记 |

为了保留每次审核和重新提交历史，保留 `t_question_correction_review_record`，但不再记录 `review_round`：

| 字段 | 含义 |
| --- | --- |
| `correction_id` | 改错记录 ID |
| `review_result` | `APPROVED` 或 `REJECTED` |
| `student_wrong_reason` | 本次审核时学生提交的错误原因 |
| `student_correct_thinking` | 本次审核时学生提交的正确思路 |
| `reviewer_id` | 审核人 |
| `reviewer_name` | 审核人姓名 |
| `review_comment` | 审核说明 |
| `create_time` | 创建时间 |

学生每次重新提交时，主表覆盖为最新提交内容，历史提交和历史审核可从审核记录和提交时间追踪；如果后续需要完整提交历史，可再扩展 `t_question_correction_submit_record`，第一版先不引入。

### 学生端错题本

错题本需要按改错状态分层展示，避免所有错题混在一起。建议使用 Tab 或分组列表：

| 层次 | 状态来源 | 展示和操作 |
| --- | --- | --- |
| 未提交改错 | 没有改错记录，或 `review_status = NOT_SUBMITTED` | 显示“提交改错”按钮 |
| 改错待审核 | `review_status = SUBMITTED` | 展示学生已提交内容，按钮置灰或仅允许查看 |
| 改错已通过 | `review_status = APPROVED` | 展示学生提交内容和老师审核意见，不再提示提交 |
| 改错被驳回 | `review_status = REJECTED` | 展示驳回原因，显示“重新提交改错”按钮 |

学生填写：

- 我错在哪里。
- 正确思路是什么。

错题详情建议增加更清晰的层次：

- 题目区：题干、选项、学生答案、正确答案。
- 解析区：标准解析。
- 改错区：当前改错状态、学生最近提交内容、老师最近审核意见。
- 历史区：历次被驳回记录和审核意见。

提交规则：

- 未提交、被驳回状态允许提交。
- 待审核状态不允许重复提交，避免覆盖老师正在审核的内容。
- 已通过状态不允许提交，除非未来新增“申请重新改错”功能。
- 重新提交后状态从 `REJECTED` 改为 `SUBMITTED`，并清空最近审核人和审核时间，保留上一条审核记录。

### 管理端改错审核

老师页面展示：

- 题干。
- 学生原答案。
- 正确答案。
- 标准解析。
- 学生提交的错误原因和正确思路。
- 老师审核结论：通过 / 不通过。
- 老师审核意见。
- 历史审核记录。

老师操作：

- 选择通过。
- 或选择不通过，并填写原因或修改建议。
- 保存审核结论。

保存后：

- 通过时，主表 `review_status` 更新为 `APPROVED`。
- 不通过时，主表 `review_status` 更新为 `REJECTED`。
- 更新审核人、审核时间和审核说明。
- 记录审核历史。

管理端筛选条件：

| 筛选项 | 含义 |
| --- | --- |
| `reviewStatus` | `SUBMITTED`、`APPROVED`、`REJECTED` |
| `studentKeyword` | 学生用户名或姓名 |
| `questionKeyword` | 题干关键字 |
| `knowledgePoint` | 知识点 |
| `submitTimeRange` | 提交时间范围 |

默认列表应优先显示 `SUBMITTED`，即待审核改错。

## 四、实施阶段

### 第一阶段：智能训练上下限

目标：解决低频知识点过度出现问题。

改动：

- 升级智能训练规则 VM。
- 实现旧 JSON 兼容读取。
- 实现上下限加权抽题。
- 管理端支持上下限配置。
- 更新默认 SQL 配置。

验收：

- 低频知识点可配置 `0-1`。
- 训练总题数仍稳定等于配置总题数。
- 配置不合理时返回明确错误。
- 旧固定题数配置仍能正常生成训练卷。

### 第二阶段：题目解析审核

目标：老师能审核并修改解析，支持 Markdown 预览，保留自动轮次审核历史。

改动：

- 新增题目审核记录表。
- 新增解析审核接口。
- 管理端新增解析审核页面。
- 解析编辑增加 Markdown 实时预览。
- 审核轮次由后端根据历史记录自动推导。
- 题目质量审核列表支持按未审核、一审、二审筛选。
- 题目详情展示审核状态。

验收：

- 老师可修改解析并保存，首次自动记为一审。
- 老师可基于一审结果继续保存，系统自动记为二审。
- 老师能在保存前预览 Markdown 渲染效果。
- 老师能直接筛选解析未审核题目。
- 题目正式解析使用最新审核版本。
- 审核历史可追溯。

### 第三阶段：知识点审核

目标：老师能结合当前等级知识点分布，审核并修改题目知识点。

改动：

- 新增知识点分布接口。
- 管理端新增知识点审核视图。
- 支持选择已有知识点和新增知识点。
- 写入知识点审核记录。
- 审核轮次由后端根据历史记录自动推导。
- 题目质量审核列表支持按知识点审核状态筛选。

验收：

- 老师可看到当前等级全部知识点分布。
- 老师可修改题目知识点。
- 老师能直接筛选知识点未审核题目。
- 审核记录保留修改前后知识点。
- 智能训练配置可复用同一分布接口。

### 第四阶段：错题改错审核

目标：学生提交改错，老师用通过 / 不通过保证改错质量，不通过后学生可重新提交。

改动：

- 新增错题改错表和审核记录表。
- 学生端错题本增加提交改错。
- 管理端增加改错审核。
- 管理端改错审核去掉审核轮次，增加通过 / 不通过操作。
- 错题本按未提交改错、改错待审核、改错已通过、改错被驳回分层展示。
- 被驳回的改错支持重新提交。

验收：

- 学生可提交错误原因和正确思路。
- 老师可选择通过或不通过。
- 通过后错题本展示为改错已通过，不再提示继续提交。
- 不通过后错题本展示为改错被驳回，并展示老师意见。
- 被驳回后学生可重新提交，重新提交后进入待审核。
- 审核记录完整保留每次结论和老师意见。

## 五、优先级建议

1. 先做智能训练上下限，直接改善训练体验。
2. 再做解析审核，因为解析质量会影响学生理解。
3. 再做知识点审核，因为它会反向影响智能训练配置质量。
4. 最后做错题改错审核，因为它同时涉及学生端、管理端和作答记录。

## 六、风险与约束

- 旧规则兼容必须做好，否则已部署环境的智能训练可能无法生成。
- 审核修改题目解析时，需要正确更新题目内容 JSON，避免破坏题干、选项和答案。
- 知识点新增需要统一命名规则，GESP 题库继续使用 `GESP{N}级/知识点`。
- 解析 Markdown 预览必须复用正式渲染组件，否则老师看到的预览和学生看到的解析可能不一致。
- 题目质量审核轮次由后端计算，不能相信前端传入值，否则并发保存时可能出现重复轮次。
- 错题改错被驳回后，要避免学生覆盖待审核内容；只有未提交和被驳回状态可提交。
- 题目解析和知识点多轮审核应保留每一轮修改前后的内容，不能只覆盖最终结果。
- 错题改错审核记录应保留每次通过 / 不通过结论，不能只看主表最新状态。

## 七、基于当前实现的修改清单

### 后端接口调整

题目质量审核：

- `QuestionReviewController` 的解析审核、知识点审核保存接口不再接收可信的 `reviewRound`；如果前端仍传入该字段，后端忽略。
- 保存解析或知识点前，后端按 `question_id + review_type` 查询历史最大轮次并自动加一。
- 题目质量审核分页接口增加 `reviewType`、`reviewStatus`、`keyword` 等筛选字段。
- 分页接口默认支持查询未审核题目，未审核判断基于 `t_question_review_record` 是否存在对应类型记录。

错题改错：

- `QuestionCorrectionController` 管理端审核接口从 `reviewRound + reviewedWrongReason + reviewedCorrectThinking` 改为 `reviewResult + reviewComment`。
- `reviewResult = APPROVED` 时，主记录状态更新为 `APPROVED`。
- `reviewResult = REJECTED` 时，主记录状态更新为 `REJECTED`，学生端可重新提交。
- 学生重新提交时仅允许当前状态为未提交或被驳回；重新提交后状态更新为 `SUBMITTED`。
- 改错审核记录表保存 `review_result`、学生当次提交内容、审核人和审核意见，不再保存审核轮次。

### 数据库调整

题目审核记录表保持 `review_round`，用于解析和知识点多轮审核。

错题改错表需要调整：

- `review_status` 枚举语义改为 `SUBMITTED`、`APPROVED`、`REJECTED`；未提交状态可由无记录表示，必要时再冗余 `NOT_SUBMITTED`。
- 删除或停用 `reviewed_wrong_reason`、`reviewed_correct_thinking` 的业务含义。
- 新增或保留 `review_comment`，作为老师通过 / 不通过意见。
- 可新增 `resubmit_count`，用于统计重新提交次数。

错题改错审核记录表需要调整：

- 删除或停用 `review_round`。
- 新增 `review_result`。
- 保留本次审核时的学生提交内容，便于追溯被驳回时学生提交了什么。

### 管理端页面调整

题目质量审核页：

- 解析审核弹窗增加 Markdown 编辑 / 预览双栏。
- 本次审核轮次显示为只读文案，例如“本次保存后为二审”。
- 列表默认筛选未审核题目。
- 筛选栏增加审核类型、审核状态、关键字。
- 列表列展示解析审核状态和知识点审核状态。

改错审核页：

- 移除审核轮次输入。
- 移除老师修订学生改错文本的表单。
- 增加通过 / 不通过选项。
- 不通过时要求填写审核意见。
- 默认筛选待审核记录。

### 学生端错题本调整

- 错题本按未提交改错、改错待审核、改错已通过、改错被驳回分层展示。
- 错题详情分为题目、解析、改错、历史四个区域。
- 未提交和被驳回状态显示提交入口。
- 待审核状态只展示已提交内容，不允许重复提交。
- 已通过状态展示通过结果和老师意见，不再显示提交入口。
