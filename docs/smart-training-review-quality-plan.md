# 智能训练与题目质量审核方案

## 背景

当前智能训练配置按知识点固定题数抽题，例如某个知识点固定抽 1 题。对于低频知识点，这会导致它们在每次训练中稳定出现，实际效果上把“不常见知识点”变成了“常见知识点”。

题库质量侧还需要补充老师审核能力：老师可以审核并修改题目解析、知识点配置，以及学生错题改错内容。这里的“审核”包含修改动作，不设计“驳回”状态。老师发现内容不合格时，直接修订为合格版本，并记录审核人、审核时间、审核前后内容和审核轮次。

## 目标

- 智能训练支持每个知识点配置抽题下限和上限。
- 低频知识点可配置为 `0-1`，避免每次训练都出现。
- 智能训练抽题在满足总题数的前提下，按知识点分布和权重随机分配。
- 老师可以审核并修改题目解析。
- 老师可以审核并修改题目知识点。
- 老师审核知识点时能看到该等级下全部知识点分布。
- 老师可以选择已有知识点，也可以新增知识点。
- 审核记录支持一审、二审等多轮记录。
- 学生错题本支持提交改错说明，老师审核时可直接修改为最终合格版本。

## 非目标

- 不做“驳回 / 重新提交”流程。
- 不要求学生端参与题目解析或知识点审核。
- 不在第一阶段重构完整题库模型。
- 不强制引入工作流引擎，先用明确状态字段和审核记录表实现。

## 一、智能训练上下限配置

### 配置结构

将当前规则：

```json
{
  "knowledgePoint": "GESP1级/循环结构",
  "questionCount": 5
}
```

升级为：

```json
{
  "knowledgePoint": "GESP1级/循环结构",
  "minCount": 2,
  "maxCount": 6,
  "weight": 105,
  "enabled": true
}
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `knowledgePoint` | 知识点名称，继续使用 `GESP{N}级/知识点` 格式 |
| `minCount` | 每次训练至少抽几题，可为 0 |
| `maxCount` | 每次训练最多抽几题 |
| `weight` | 分配剩余题数时使用的权重，默认可取该知识点题库数量 |
| `enabled` | 是否启用该知识点 |

兼容规则：

- 如果旧配置只有 `questionCount`，则读取时视为 `minCount = questionCount`、`maxCount = questionCount`。
- 保存后统一写入新结构。

### 抽题算法

1. 读取当前科目智能训练配置。
2. 过滤 `enabled = false` 的知识点。
3. 校验每条规则：`0 <= minCount <= maxCount`。
4. 先按 `minCount` 为每个知识点分配基础题数。
5. 计算剩余题数：`remaining = totalQuestionCount - sum(minCount)`。
6. 在未达到 `maxCount` 的知识点中，按 `weight` 加权随机分配剩余题数。
7. 如果某知识点可用题数不足，按实际可用题数降级，并将缺口重新分配给其他知识点。
8. 如果所有可用题都不足以满足总题数，返回明确错误。
9. 每个知识点内仍按随机题目抽取。

### 默认配置建议

按知识点分布生成默认上下限：

| 分布数量 | 建议区间 |
| ---: | --- |
| 1-3 | `0-1` |
| 4-10 | `0-2` |
| 11-30 | `1-3` |
| 31-80 | `1-5` |
| 80 以上 | `2-6` |

这样低频知识点不会每次出现，高频知识点仍能保持更高出现概率。

### 涉及改动

- 后端 `SmartTrainingRuleVM` 增加 `minCount`、`maxCount`、`weight`、`enabled`。
- 后端智能训练抽题逻辑从固定题数改为上下限加权分配。
- 后端配置校验从“规则题数合计等于总题数”改为：
  - `sum(minCount) <= totalQuestionCount`
  - `sum(maxCount) >= totalQuestionCount`
  - 每条规则 `minCount <= maxCount`
- 管理端智能训练配置页增加上下限、权重、启用开关。
- SQL 默认配置从固定题数 JSON 改为上下限 JSON。

## 二、题目解析与知识点审核

### 核心原则

审核不是“通过 / 驳回”，而是“老师把内容修订到可用状态”。每次审核都保存审核前内容、审核后内容、审核人和轮次。

### 新增表：题目审核记录

建议新增 `t_question_review_record`：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `question_id` | 题目 ID |
| `review_type` | `ANALYSIS` 或 `KNOWLEDGE_POINT` |
| `review_round` | 审核轮次，例如 1、2 |
| `before_value` | 审核前内容 |
| `after_value` | 审核后内容 |
| `reviewer_id` | 审核老师 ID |
| `reviewer_name` | 审核老师姓名 |
| `review_comment` | 审核说明 |
| `create_time` | 记录创建时间 |

题目表可增加质量状态字段：

| 字段 | 含义 |
| --- | --- |
| `analysis_review_round` | 解析已完成的最高审核轮次 |
| `analysis_review_user_id` | 最近解析审核人 |
| `analysis_review_time` | 最近解析审核时间 |
| `knowledge_review_round` | 知识点已完成的最高审核轮次 |
| `knowledge_review_user_id` | 最近知识点审核人 |
| `knowledge_review_time` | 最近知识点审核时间 |

不增加 `REJECTED` 状态。需要区分时使用：

- `UNREVIEWED`：未审核。
- `REVIEWED_ONCE`：一审完成。
- `REVIEWED_TWICE`：二审完成。

这些状态可由审核轮次推导，也可以冗余存储便于筛选。

### 解析审核

老师页面展示：

- 题干。
- 选项。
- 正确答案。
- 当前解析。
- 可编辑解析。
- 审核历史。

老师操作：

- 编辑解析。
- 填写审核说明。
- 保存为一审结果。
- 保存为二审结果。

保存后：

- 将修订后的解析写回题目内容 JSON。
- 写入 `t_question_review_record`。
- 更新题目解析审核轮次和审核人。

### 知识点审核

老师页面展示：

- 题目内容。
- 当前知识点。
- 当前等级下全部知识点分布数量。
- 已有知识点选择器。
- 新增知识点输入。
- 审核历史。

老师操作：

- 选择已有知识点。
- 或新增知识点。
- 保存为一审或二审结果。

保存后：

- 将修订后的知识点写回 `t_question.knowledge_point`。
- 写入 `t_question_review_record`。
- 更新题目知识点审核轮次和审核人。

### 知识点分布接口

新增或扩展后台接口：

```text
POST /api/admin/question/knowledgePointDistribution/{subjectId}
```

返回：

```json
[
  {
    "knowledgePoint": "GESP1级/循环结构",
    "questionCount": 105,
    "reviewedCount": 80,
    "unreviewedCount": 25
  }
]
```

用途：

- 审核知识点时提供全量分布参考。
- 智能训练配置时也可复用。

## 三、错题改错审核

### 核心原则

学生提交改错内容，老师审核时可以直接修改。系统保存学生原始提交和老师修订后的最终版本，不设计驳回和重新提交。

### 新增表：错题改错记录

建议新增 `t_question_correction_record`：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `user_id` | 学生 ID |
| `question_id` | 题目 ID |
| `exam_paper_answer_id` | 试卷作答 ID |
| `customer_answer_id` | 题目作答 ID |
| `student_wrong_reason` | 学生提交的错误原因 |
| `student_correct_thinking` | 学生提交的正确思路 |
| `reviewed_wrong_reason` | 老师修订后的错误原因 |
| `reviewed_correct_thinking` | 老师修订后的正确思路 |
| `review_status` | `SUBMITTED`、`REVIEWED_ONCE`、`REVIEWED_TWICE` |
| `reviewer_id` | 最近审核老师 ID |
| `reviewer_name` | 最近审核老师姓名 |
| `review_comment` | 老师审核说明 |
| `submit_time` | 学生提交时间 |
| `review_time` | 最近审核时间 |

如果需要完整多轮记录，可再加 `t_question_correction_review_record`：

| 字段 | 含义 |
| --- | --- |
| `correction_id` | 改错记录 ID |
| `review_round` | 审核轮次 |
| `before_wrong_reason` | 审核前错误原因 |
| `before_correct_thinking` | 审核前正确思路 |
| `after_wrong_reason` | 审核后错误原因 |
| `after_correct_thinking` | 审核后正确思路 |
| `reviewer_id` | 审核人 |
| `reviewer_name` | 审核人姓名 |
| `review_comment` | 审核说明 |
| `create_time` | 创建时间 |

### 学生端错题本

错题卡片增加：

- 提交改错。
- 查看改错。
- 查看老师修订版。

学生填写：

- 我错在哪里。
- 正确思路是什么。

状态展示：

- 未提交改错。
- 已提交，待老师审核。
- 一审完成。
- 二审完成。

### 管理端改错审核

老师页面展示：

- 题干。
- 学生原答案。
- 正确答案。
- 标准解析。
- 学生提交的错误原因和正确思路。
- 老师可编辑的最终错误原因和正确思路。
- 历史审核记录。

老师操作：

- 保存一审。
- 保存二审。

保存后：

- 写入老师修订版。
- 更新审核状态和审核人。
- 记录审核历史。

## 四、实施阶段

### 第一阶段：智能训练上下限

目标：解决低频知识点过度出现问题。

改动：

- 升级智能训练规则 VM。
- 实现旧 JSON 兼容读取。
- 实现上下限加权抽题。
- 管理端支持上下限配置。
- 更新默认 SQL 配置。

验收：

- 低频知识点可配置 `0-1`。
- 训练总题数仍稳定等于配置总题数。
- 配置不合理时返回明确错误。
- 旧固定题数配置仍能正常生成训练卷。

### 第二阶段：题目解析审核

目标：老师能审核并修改解析，保留审核历史。

改动：

- 新增题目审核记录表。
- 新增解析审核接口。
- 管理端新增解析审核页面。
- 题目详情展示审核状态。

验收：

- 老师可修改解析并保存为一审。
- 老师可基于一审结果继续保存二审。
- 题目正式解析使用最新审核版本。
- 审核历史可追溯。

### 第三阶段：知识点审核

目标：老师能结合当前等级知识点分布，审核并修改题目知识点。

改动：

- 新增知识点分布接口。
- 管理端新增知识点审核视图。
- 支持选择已有知识点和新增知识点。
- 写入知识点审核记录。

验收：

- 老师可看到当前等级全部知识点分布。
- 老师可修改题目知识点。
- 审核记录保留修改前后知识点。
- 智能训练配置可复用同一分布接口。

### 第四阶段：错题改错审核

目标：学生提交改错，老师审核并修订为最终合格版本。

改动：

- 新增错题改错表和审核记录表。
- 学生端错题本增加提交改错。
- 管理端增加改错审核。
- 错题本展示审核状态和老师修订版。

验收：

- 学生可提交错误原因和正确思路。
- 老师可直接修改学生提交内容。
- 审核后学生可看到老师修订版。
- 一审、二审记录完整保留。

## 五、优先级建议

1. 先做智能训练上下限，直接改善训练体验。
2. 再做解析审核，因为解析质量会影响学生理解。
3. 再做知识点审核，因为它会反向影响智能训练配置质量。
4. 最后做错题改错审核，因为它同时涉及学生端、管理端和作答记录。

## 六、风险与约束

- 旧规则兼容必须做好，否则已部署环境的智能训练可能无法生成。
- 审核修改题目解析时，需要正确更新题目内容 JSON，避免破坏题干、选项和答案。
- 知识点新增需要统一命名规则，GESP 题库继续使用 `GESP{N}级/知识点`。
- 错题改错审核不做驳回后，老师需要承担最终文本质量，管理端编辑体验要足够顺手。
- 多轮审核应保留每一轮修改前后的内容，不能只覆盖最终结果。
