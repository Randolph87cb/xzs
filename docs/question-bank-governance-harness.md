# 题库去重与解析质量治理 Harness

## 目标

建立一套可重复执行、可验收的题库治理流程，使 active 题库按 GESP/CSP 原题来源唯一，并逐题提升解析质量。

最终状态：

- active 题库中同一原题来源只保留一题，例如 `2025-03 GESP 1级 第1-25题` 每题只出现一次。
- 每道 active 题都有经过人工确认的解析。
- 解析内容质量较好，能解释正确选项、排除主要干扰项，并与题干、选项和原题语境一致。
- 重复清理、解析治理、回写、同步和验收都有清单或记录可追溯。

## 适用范围

适用于 GESP/CSP 客观题题库的重复题治理、解析质量治理、Markdown 题源回写、Fly 远端同步和验收统计。

不适用于：

- 非 GESP/CSP 来源题目的大规模内容改写。
- 未经人工确认的解析批量生成。
- 与题库治理无关的后端、前端、部署或数据库结构改造。
- 仅凭题干相似度直接合并不同原题来源的题目。

## 输入

开始每批任务前需要准备：

- 当前 active 题库统计：总题数、暂无解析题数、import key 重复数、短解析数量、模板化解析数量。
- 原题来源信息：考试名称、年月、级别或组别、题号、题干、选项、答案。
- Markdown 题源文件及其题目定位。
- Fly 远端数据库连接方式和只读/写入权限边界。
- 待审清单：重复候选、短解析候选、模板化解析候选、需要人工确认的题目。
- 上一批次的验收结果和未解决问题。

已知基线：

- 远端 Fly active 已从 4249 题清理为 2125 题。
- `4250-6373` 共 2124 道 `<p>暂无解析</p>` 占位重复题已备份并软删除。
- 当前远端 active 题 2125，暂无解析 0，import key 重复 0。
- 仍存在短解析和模板化解析，需要继续逐题治理。

## 硬规则

- 解析内容必须逐题人工确认，不允许用程序、脚本、模型批量生成。
- 禁止用规则替换、模板替换、批量改写等方式重写解析。
- 自动化只允许用于查重复、查引用、生成待审清单、同步人工确认结果、验收统计。
- 重复题定义按原题来源唯一，不是单纯题干相似唯一。
- 如果题干相似但来源、题号、答案或语境不同，必须人工确认后再决定是否合并或保留。
- 主线程负责拆任务、调度、验收和 Git；实现 subagent 负责实际改动；验证 subagent 独立验收。

## 执行流程

### 1. 启动与定界

主线程确认本轮目标、批次范围、允许改动文件、数据环境、是否需要同步 Fly、是否需要 Git 操作。

检查点：

- 批次题目范围明确。
- 修改范围明确。
- 验收方式明确。
- Git 策略明确。

### 2. 生成待审清单

自动化脚本或查询只生成候选清单，不直接改解析内容。

清单至少包含：

- 题目 ID。
- 原题来源。
- 题号。
- 题干摘要。
- 当前解析。
- 问题类型：重复候选、短解析、模板化解析、来源缺失、引用异常。
- Markdown 定位。

检查点：

- 清单可追溯到具体题目和源文件。
- 候选原因可复核。
- 未把候选直接当作结论。

### 3. 小批次人工治理

每批处理 10-25 题。批次过大时必须拆分。

单题流程：

1. 人工核对原题来源、题干、选项和答案。
2. 人工判断是否为重复题，判断依据为原题来源唯一。
3. 人工一审解析内容，确认解释是否正确、完整、无误导。
4. 必要时人工重写或补充解析。
5. 人工二审确认解析与题目一致。
6. 回写 Markdown 题源。
7. 同步人工确认结果到 Fly。
8. 记录处理结果和审查人结论。

检查点：

- 每题都有人工一审和人工二审结论。
- 每题都有明确处理状态：保留、软删除、改解析、无需修改、待确认。
- 解析变更能从 Markdown 追溯到数据库同步结果。

### 4. 同步与独立验证

实现 subagent 完成回写和同步后，验证 subagent 独立读取 Markdown、远端统计和抽样题目，不依赖实现 subagent 的口头结论。

验证内容：

- active 总题数是否符合预期。
- 暂无解析是否为 0。
- import key 是否无重复。
- 本批题目是否按原题来源唯一。
- 本批 Markdown 与 Fly 内容是否一致。
- 本批解析是否都有人工确认记录。
- 是否仍有新增短解析、模板化解析或明显错误解析。

检查点：

- 验证结果明确通过或不通过。
- 不通过项能定位到题目 ID、来源和文件位置。
- 主线程根据验证结果决定继续、返工或暂停。

### 5. 汇总与推进

主线程汇总每批结果，更新剩余清单和风险项，再决定下一批范围。

检查点：

- 本批产出已归档。
- 失败项已进入下一批或阻塞清单。
- 下一批范围不超过 25 题。

## 角色分工

主线程：

- 定义批次目标、边界和完成标准。
- 分配实现 subagent 和验证 subagent。
- 汇总结果、处理阻塞、决定是否继续。
- 负责 Git 操作和最终验收。

实现 subagent：

- 只在指定范围内执行实际改动。
- 生成或更新待审清单。
- 根据人工确认结果回写 Markdown。
- 按批准步骤同步 Fly。
- 返回改动范围、执行命令、统计结果和未解决问题。

验证 subagent：

- 独立读取实际文件和远端数据。
- 复核重复定义、解析质量、Markdown 与 Fly 一致性。
- 输出通过/不通过结论和可定位问题清单。

人工审查：

- 逐题确认原题来源。
- 逐题一审、二审解析。
- 对是否重复、是否保留、如何改解析给出结论。

自动化：

- 查重复。
- 查引用。
- 生成待审清单。
- 同步人工确认结果。
- 生成验收统计。

## 产出

每批至少产出：

- 待审清单。
- 人工一审记录。
- 人工二审记录。
- Markdown 回写变更。
- Fly 同步记录。
- 独立验证报告。
- 剩余问题清单。

最终产出：

- active 题库来源唯一性验收结果。
- 解析质量治理验收结果。
- 题库统计汇总。
- 可复用的批次处理记录。

## 失败处理

信息不足：

- 停止处理相关题目。
- 标记为待确认。
- 向主线程说明缺失信息，不猜测来源或解析。

人工审查不一致：

- 不同步 Fly。
- 保留原内容或回到上一稳定版本。
- 交由主线程决定是否补充第三方复核。

同步失败：

- 停止本批后续同步。
- 保留已执行命令、错误信息和影响范围。
- 验证 Markdown 与 Fly 是否出现不一致。

验证不通过：

- 不进入下一批。
- 按题目 ID 和文件位置返工。
- 返工后重新执行独立验证。

发现范围外问题：

- 只记录问题。
- 不顺手修改。
- 由主线程决定是否另起批次或另建任务。

## Git 策略

- 默认由主线程负责 `git add`、`git commit`、`git push`。
- 实现 subagent 和验证 subagent 不执行 Git 提交或推送。
- 执行会改变仓库状态的 Git 命令前，必须先查看当前状态。
- 提交信息如无特殊要求使用中文。
- 如果工作区存在无关改动，不覆盖、不回滚，只在报告中说明。
