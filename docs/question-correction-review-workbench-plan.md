# 改错审核工作台、学生改错双栏与 AI 手动预审方案

## 背景与现状

- 老师端 `frontend/apps/admin/src/views/question/QuestionCorrectionReviewView.vue` 当前是列表页点击“审核”后打开弹窗。弹窗里纵向展示题目上下文、AI 预审、学生改错、审核表单和历史记录，题目稍长就需要大量滚动。
- 学生端 `frontend/apps/student/src/views/question/QuestionErrorView.vue` 已经去掉重新提交弹窗，但页面仍是左侧宽表格 + 右侧纵向详情。左侧错题列表占用空间过大，右侧题目、解析、改错表单和历史仍然太长。
- 管理端 AI 预审已有自动触发链路，学生提交或重新提交后会自动按班级负责老师配置执行。但管理端目前缺少手动触发入口，不方便验证本地管理员 API Key 是否可用，也不方便补跑历史待审核记录。
- AI 预审只能作为辅助意见，不能自动通过或驳回，最终仍必须由老师确认。

## 结论

本次推荐拆成三个独立改造：先补 AI 手动触发接口和按钮，方便开发期验证；再把老师端审核从弹窗改成“队列 + 双栏工作台 + 连续审批”；最后把学生端错题本改成“紧凑错题队列 + 双栏详情”。题目上下文继续复用 `QuestionCorrectionContext`，保证题面、选项、解析、学生答案和正确答案在老师端与学生端一致。

## 需求拆解

### 1. 老师端改错审核改为双栏连续审批工作台

- 当前现状：
  - 审核详情在 `el-dialog width="860px"` 中展示，题目、解析、改错、AI 预审和审核表单全部纵向堆叠。
  - 老师每审核一题都要打开弹窗、滚动、提交、关闭，再回到列表选择下一题。
- 判断：
  - 审核是一种连续处理队列的工作，不适合弹窗单点操作。
  - 数据已经足够支持工作台化，不需要改变主查询模型。
- 修改方案：
  - 页面顶部保留筛选区：状态、班级、AI 状态、查询按钮和“AI 批量预审”按钮。
  - 左侧改为审核队列，宽度约 300-360px，展示题干摘要、学生、提交时间、AI 状态和审核状态。
  - 主工作区改为左右两栏：
    - 左栏：题目上下文，复用 `QuestionCorrectionContext`，展示题面、选项、学生选择、正确答案、解析，独立滚动。
    - 右栏：学生改错、AI 预审建议、审核结果、审核意见、审核历史，审核表单保持在右栏靠上位置。
  - 增加连续审批：
    - “保存并下一题”作为主按钮。
    - “仅保存”作为次按钮。
    - 保存成功后自动选择当前队列下一条待审核记录；当前页没有下一条时刷新列表并选中第一条。
  - 驳回时审核意见必填；通过时可选填。
- 影响范围：
  - `frontend/apps/admin/src/views/question/QuestionCorrectionReviewView.vue`
  - 可选拆分组件：`CorrectionReviewQueue.vue`、`CorrectionReviewWorkspace.vue`、`CorrectionReviewDecisionPanel.vue`
- 验证方案：
  - 进入改错审核页后无需弹窗即可看到第一条详情。
  - 长题目只让左栏滚动，右栏审核表单不被推到屏幕底部。
  - 点击不同队列项能同步切换题目、学生改错和 AI 预审。
  - 保存并下一题后自动跳到下一条待审核记录。

### 2. 学生端错题本改为紧凑列表 + 双栏详情

- 当前现状：
  - 左侧表格列较多，占据接近半屏空间。
  - 右侧题目上下文、解析、改错表单和历史纵向堆叠，学生仍要频繁上下滚动。
- 判断：
  - 学生的主任务是选中一题并修改改错内容，错题列表只需要承担定位作用。
  - 桌面端应把题目上下文和改错表单并排展示，减少记忆负担。
- 修改方案：
  - 左侧错题列表改为紧凑队列：
    - 宽度约 280-340px。
    - 用题干摘要、状态 tag、学科和做题时间替代表格多列。
    - 保留状态 tabs、刷新和分页。
  - 右侧详情改为双栏：
    - 左栏：题目、选项、学生答案、正确答案、解析。
    - 右栏：老师驳回意见、改错表单或只读改错内容、历史记录。
  - 老师驳回意见放在右栏顶部，使用醒目的红色/警告提示块，完整换行展示。
  - 未提交和被驳回状态允许编辑；待审核和已通过状态只读。
  - 窄屏降级为单列，顺序为错题队列、题目上下文、改错表单。
- 影响范围：
  - `frontend/apps/student/src/views/question/QuestionErrorView.vue`
  - 可选拆分组件：`QuestionErrorQueue.vue`、`QuestionCorrectionPanel.vue`
- 验证方案：
  - 桌面端题目上下文和改错表单能同屏并排看到。
  - 左侧队列不因题干过长撑宽页面。
  - 驳回状态下，学生首先看到老师审核意见，再修改改错内容。
  - 四种状态：未提交、待审核、已通过、被驳回的编辑权限保持正确。

### 3. AI 预审支持单题和批量手动触发

- 当前现状：
  - 自动预审只在学生提交或重新提交后触发。
  - 管理端没有“当前题重跑 AI”或“批量补跑 AI”的入口。
  - `QuestionCorrectionAiReviewService.preReview(correctionId, triggerType)` 已经是可复用的 public 方法。
- 判断：
  - 手动触发应复用现有 OpenAI-compatible 调用链，只补管理端接口、权限和 UI。
  - 手动触发仍按改错所属班级负责老师的 AI 配置执行，保持与自动触发语义一致。
- 修改方案：
  - 后端新增接口：
    - `POST /api/admin/questionCorrection/ai/review/{id}`：手动触发单条 AI 预审，`triggerType = MANUAL_SINGLE`。
    - `POST /api/admin/questionCorrection/ai/review/batch`：按当前筛选条件批量触发，`triggerType = MANUAL_BATCH`。
  - 单题触发：
    - 老师只能触发自己负责班级的记录。
    - 管理员可以触发全量记录。
    - 返回最新 AI 记录或触发结果，前端刷新当前详情。
  - 批量触发：
    - 默认只处理 `reviewStatus = SUBMITTED` 的记录。
    - 按当前筛选条件执行，第一版设置单次上限 50 条，避免误消耗大量 API。
    - 返回 `acceptedCount`、`skippedCount`、`failedCount` 和失败摘要。
  - 前端入口：
    - 顶部增加“AI 批量预审”按钮。
    - 右栏 AI 区域增加“AI 预审当前题/重新预审”按钮。
    - 触发后显示 loading，结束后刷新当前详情和队列 AI 状态。
  - 本地测试注意：
    - 用户已在本地开发环境配置管理员 API Key。
    - 如果测试记录所在班级的负责老师不是该管理员，现有设计会按班级负责老师配置执行，可能导致“未配置 AI”。测试前应确认被测班级负责老师是当前管理员。
- 影响范围：
  - `source/xzs/src/main/java/com/mindskip/xzs/controller/admin/QuestionCorrectionController.java`
  - `source/xzs/src/main/java/com/mindskip/xzs/service/QuestionCorrectionAiReviewService.java`
  - `frontend/packages/api-client/src/adminOperations.ts`
  - `frontend/apps/admin/src/views/question/QuestionCorrectionReviewView.vue`
- 验证方案：
  - 单题触发成功后新增 `MANUAL_SINGLE` AI 记录，页面展示最新建议。
  - 批量触发成功后，符合筛选条件的多条待审核记录新增 `MANUAL_BATCH` AI 记录。
  - AI 建议不改变 `review_status`。
  - API Key 错误或模型调用失败时，页面显示失败原因，不影响人工审核。

## 推荐视觉结构

### 老师端审核工作台

- 顶部：筛选条件 + 查询 + AI 批量预审。
- 左侧：待审核队列，作为导航和处理进度。
- 中间：题目上下文，重点展示题面、选项、解析、学生答案和正确答案。
- 右侧：学生改错、AI 建议、审核结果、审核意见、保存并下一题。

### 学生端错题本

- 左侧：紧凑错题队列，只保留定位所需信息。
- 中间：题目上下文。
- 右侧：老师意见、改错表单、提交状态和历史。

## 执行顺序

1. 后端增加 AI 单题和批量手动触发接口，补权限校验和测试。
2. 管理端接入 AI 手动触发按钮，先保证可以用本地管理员 API Key 做真实测试。
3. 管理端审核页改为队列 + 双栏工作台，并实现保存并下一题。
4. 学生端错题本改为紧凑队列 + 双栏详情。
5. 做浏览器截图验收：老师端桌面、学生端桌面、学生端窄屏。
6. 做真实 AI 调用验收：单题触发、批量小样本触发、失败提示。

## 风险与待确认

- 批量 AI 预审的“全量”推荐解释为“当前筛选条件下的待审核记录”，第一版单次上限 50 条。
- 手动 AI 触发仍按班级负责老师配置执行；如果要新增“用当前管理员配置强制测试”的特殊入口，需要单独确认，因为它会偏离自动触发规则。
- 连续审批自动跳转可能让老师错过保存结果，右栏顶部需要保留短暂的上一条保存结果提示。
- 如果题库本身没有解析，界面只能显示“暂无解析”，需要另行补题库数据。
